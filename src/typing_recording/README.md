# Typewriter Key Classification using Acceleration Trigger and Audio Features

## 概要

このプログラム群は、タイプライターのキー押下を加速度センサで検出し、そのタイミングをトリガとして打鍵音を録音し、録音した音データから押されたキーを推定するための実験用プログラムです。

タイプライターは、一般的なPCキーボードのように電気的なキー入力信号を直接取得できません。  
そのため、本プロジェクトでは以下の流れでキー入力の推定を行います。

1. タイプライターのキャリッジ部に加速度センサを取り付ける
2. キー押下時に発生する機械的な衝撃を検出する
3. 加速度トリガをPCへ送信する
4. PC側でトリガ前後の音声を録音する
5. 録音した打鍵音から機械学習によりキーを分類する

本実験では、`D`, `E`, `H`, `L`, `O`, `R`, `W` の7キーを対象とし、各キー30サンプルずつ録音して分類精度を評価しました。

## 目的

この実験の目的は、タイプライターの打鍵音にキーごとの違いが含まれているかを確認し、機械学習によってキー推定が可能かを検証することです。

目視で波形を確認した限りでは、キーごとの違いは明確には分かりませんでした。  
しかし、MFCCや周波数帯域エネルギーなどの音響特徴量を用いることで、波形を直接見ただけでは分からない差分を利用できる可能性があります。

## 加速度トリガ側仕様

Arduino側では、BMX055のZ軸加速度のみを1kHz周期で取得し、キー押下と判断した場合のみPCへ `1` を送信します。

| 項目 | 内容 |
|---|---|
| 取得軸 | Z軸加速度 |
| 取得周期 | 1kHz |
| 加速度レンジ | ±2g |
| I2Cクロック | 400kHz |
| 押下検出条件 | `|Z - baseline| > 0.5g` |
| 起動時キャリブレーション | 500サンプル、約0.5秒 |
| 再検出無視時間 | 600ms |
| シリアル送信内容 | 押下検出時のみ `1` |

### 押下検出条件

押下検出条件は以下です。

```text
|Z - baseline| > 0.5g
```

`baseline` は、起動直後にキーを押していない状態で取得したZ軸加速度の平均値です。

起動直後の500サンプル、約0.5秒間を基準値取得期間とし、この期間中は押下検出を行いません。

### 再検出無視時間

キー押下後は、キャリッジやタイプライター本体に振動が残ります。  
単純なしきい値判定だけでは、1回の押下を複数回検出する可能性があります。

そのため、押下検出後は一定時間、次の検出を無視します。

```text
IGNORE_PERIOD_MS = 600
```

現在の実装では600msに設定しています。

なお、コードコメント内に300msという記述が残っている場合がありますが、実際の定数値は600msです。  
連続入力の反応性を上げたい場合は300ms程度に短くすることもできますが、誤検出や二重検出とのバランスを見ながら調整します。

## 音声録音側仕様

PC側の録音プログラムは、Arduinoからシリアルで `1` を受信したタイミングをトリガとして、トリガ前後の音声をWAVファイルに保存します。

| 項目 | 内容 |
|---|---|
| 入力デバイス | ZOOM AMS-44 ASIO Driver |
| サンプリング周波数 | 96kHz |
| 入力チャンネル数 | 4ch |
| 保存チャンネル | 1ch目のみ |
| 保存形式 | WAV / PCM 24bit |
| 保存先 | `recorded_triggers` |
| トリガ | Arduinoからの `1` |
| 音声保持方式 | リングバッファ |

### リングバッファを使う理由

トリガを受信してから録音を開始すると、打鍵音の先頭が欠ける可能性があります。

そこで、PC側では常に音声入力を行い、直近数秒分の音声をリングバッファに保持します。  
Arduinoから `1` を受信したら、リングバッファからトリガ前の音声を取り出し、さらにトリガ後の音声が入るまで待ってからWAV保存します。

これにより、トリガ受信の遅延があっても、打鍵音の立ち上がりを保存しやすくしています。

### 録音範囲

録音範囲はPythonコード内の以下の定数で調整します。

```python
PRE_TRIGGER_SEC = 0.015
POST_TRIGGER_SEC = 0.060
```

現在の設定では、トリガ前15ms、トリガ後60msを保存します。

当初は前後100msを想定していましたが、打鍵音の主要部分が短時間に集中するため、現在は保存データ量と分類用データの扱いやすさを考慮して短めにしています。

必要に応じて、以下のように変更できます。

```python
PRE_TRIGGER_SEC = 0.100
POST_TRIGGER_SEC = 0.100
```

### 4ch入力して1chのみ保存する理由

使用しているZOOM AMS-44 ASIO Driverでは、1ch入力として開くよりも、4ch入力として開いた方が安定する場合があります。

そのため、録音プログラムでは4ch入力で音声ストリームを開き、WAV保存時に指定した1chのみを切り出します。

```text
INPUT_CHANNELS = 4
SAVE_CHANNEL_INDEX = 0
```

この設計により、ASIOドライバ側のチャンネル構成に合わせつつ、分類用データは1ch音声として保存できます。

各キー30サンプルずつ、合計210サンプルを使用しました。

## 学習側仕様

学習プログラムでは、各WAVファイルから音響特徴量を抽出し、SVMで分類します。

| 項目 | 内容 |
|---|---|
| 分類対象 | `D`, `E`, `H`, `L`, `O`, `R`, `W` |
| サンプル数 | 各キー30個 |
| 合計サンプル数 | 210個 |
| 特徴量数 | 194 |
| 分類器 | SVM |
| カーネル | RBF |
| 標準化 | `StandardScaler` |
| テストデータ割合 | 20% |
| クロスバリデーション | 5-fold StratifiedKFold |
| 保存モデル | `typewriter_key_classifier.joblib` |

### 使用する主な特徴量

波形を目視してもキーごとの差が分かりにくいため、音声波形をそのまま分類するのではなく、音響特徴量に変換して分類しています。

使用している主な特徴量は以下です。

- MFCC
- MFCCの時間変化量
- RMS
- Zero Crossing Rate
- Spectral Centroid
- Spectral Bandwidth
- Spectral Rolloff
- Spectral Flatness
- FFTによる周波数帯域エネルギー

これらの特徴量について、時間方向の平均、標準偏差、最大、最小などを計算し、固定長の特徴ベクトルとしてSVMに入力します。

---

## SVMを使用する理由

今回のデータ数は、7キー×30サンプルで合計210サンプルです。

この規模では、深層学習モデルを使うにはデータが少なく、過学習しやすいと考えられます。

そのため、まずは以下の構成を採用しました。

```text
音響特徴量抽出 + 標準化 + SVM
```

SVMは少量データでも比較的扱いやすく、特徴量設計の効果を確認しやすいため、初期実験に向いています。

## 結果の考察

今回の結果から、タイプライターの打鍵音にはキーごとの違いが含まれており、音響特徴量と機械学習によって分類できる可能性が確認できました。

ただし、各キー30サンプルという少量データであるため、現時点の正答率はあくまで初期実験の結果です。

特に以下の点には注意が必要です。

- テストデータは各キー6個のみであり、1個の誤分類で評価値が大きく変わる
- 録音条件が変わった場合の汎化性能は未確認
- キーを順番にまとめて録音している場合、録音時間帯やキャリッジ位置の影響を学習している可能性がある
- `L` と `O` の分離が弱い
- `O` が誤分類の吸い込み先になっている

そのため、現時点では

```text
同一条件下であれば、7キー分類はある程度可能
```

という段階です。

実用化には、録音条件を変えた追加データによる検証が必要です。

---

## 今後の改善方針

### データ数を増やす

各キー30サンプルでは少ないため、まずは各キー100サンプル以上に増やすことを目標にします。

特に、誤分類が多かった以下のキーを重点的に増やします。

```text
L, O, R, W
```

### 録音順をランダム化する

キーごとに30回まとめて録音すると、キー音以外の要因を学習してしまう可能性があります。

例えば、以下のような要因です。

- 録音時刻
- 打鍵の慣れ
- キャリッジ位置
- マイク位置の微妙な変化
- タイプライター本体の振動状態

そのため、今後は以下のようにキー順を混ぜて録音します。

```text
D, E, H, L, O, R, W
W, D, O, H, E, R, L
...
```

### 時間分割特徴量を追加する

現在の特徴量は、音声全体に対して平均、標準偏差、最大、最小などを計算しています。  
そのため、時間方向の構造が一部失われています。

タイプライター音には以下のような時間的な構造がある可能性があります。

```text
キー押下の衝撃音
タイプバーが動く音
印字・打撃音
キャリッジ側の振動
残響
```

そのため、音声を複数区間に分割し、それぞれで特徴量を抽出すると、`L` と `O` の分離が改善する可能性があります。

例：

```text
0〜20ms
20〜40ms
40〜75ms
```

### 加速度レンジの見直し

押下時の加速度が±2gレンジの上限付近に達しているため、加速度波形を詳細に解析する場合は±4gまたは±8gレンジを検討します。

ただし、現在は加速度をトリガ用途に限定しているため、押下検出だけであれば±2gのままでも問題ありません。

---

## 現時点の結論

今回の実験では、タイプライターのキー押下を加速度センサで検出し、そのタイミングで録音した打鍵音から、7キー分類を行いました。

結果として、以下の精度が得られました。

```text
Cross Validation accuracy: 0.767 ± 0.059
Test accuracy: 0.833
```

この結果から、波形を目視しただけでは違いが分からない場合でも、MFCCや周波数帯域エネルギーなどの音響特徴量を用いることで、キーごとの差を利用できる可能性が示されました。

現時点では実験段階ですが、タイプライターを音響的にキーボード化するという方向性には十分な見込みがあります。

今後は、データ数の増加、録音条件のランダム化、時間分割特徴量の追加により、分類精度と汎化性能の向上を目指します。
