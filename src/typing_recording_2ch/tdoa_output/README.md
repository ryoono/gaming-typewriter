# タイプライター打鍵音 TDoA解析結果メモ

## メモ
TDoAだけでは打鍵箇所を判定するのは難しい（少なくとも現状では困難）。  
ただし、左寄りか右寄りかなどの傾向はでている。  
そのため、音響解析とTDoA結果の両方をSVMで学習する方針とする。

## 概要

このメモは、`batch_typewriter_tdoa.py`の結果である`results.csv` に保存したTDoA解析結果の読み方と、今回の録音結果をまとめたものです。

今回のデータは、タイプライター上段のキーを左から右へ、各2回ずつ録音したものです。

```text
Q, Q, W, W, E, E, R, R, T, T, Y, Y, U, U, I, I, O, O, P, P
```

マイク配置は以下の前提です。

- CH1：左側マイク
- CH2：右側マイク
- マイク間距離：約21cm
- サンプリング周波数：96kHz

## 符号の読み方

TDoAの値は、以下の定義です。

```text
Δt = t_CH2 - t_CH1
```

したがって、符号の意味は以下です。

| Δt | 意味 | 解釈 |
|---:|---|---|
| 正の値 | CH1が先に音を拾った | 左側のキー寄り |
| 0付近 | CH1/CH2がほぼ同時 | 中央付近 |
| 負の値 | CH2が先に音を拾った | 右側のキー寄り |

96kHz録音なので、1サンプルは約0.0104msです。

また、マイク間距離21cm、音速343m/sとすると、物理的に取り得る最大時間差は以下です。

```text
0.21 m / 343 m/s ≒ 0.612 ms
```

そのため、±0.612ms付近に張り付く値は、物理上限に近く、相互相関が端を選んでいる可能性もあります。

## CSV列の読み方

| 列名 | 意味 |
|---|---|
| `file` | 解析したWAVファイル名 |
| `fs_Hz` | サンプリング周波数 |
| `duration_s` | WAVの長さ |
| `second_event_peak_ms` | 2つ目イベントの代表ピーク位置 |
| `tail_peak_ms` | 2つ目イベント後半側の高域ピーク位置 |
| `tdoa_window_start_ms` / `tdoa_window_end_ms` | TDoA計算に使った短い窓 |
| `tdoa_raw_ms_CH2_minus_CH1` | 補正なしのTDoA値。基本的にはこの値を見る |
| `first_event_tdoa_ms_CH2_minus_CH1` | 1つ目イベントで計算したTDoA。ゼロ点確認用 |
| `tdoa_corrected_ms` | 1つ目イベントでゼロ補正したTDoA |
| `best_corr` | 相互相関の最大値。TDoA推定の信頼度目安 |
| `energy_CH1_over_CH2_dB` | TDoA計算窓でのCH1/CH2のエネルギー比 |
| `judgement_raw` | raw値に基づく左右判定 |
| `judgement_corrected` | corrected値に基づく左右判定 |

## 今回は raw Δt を主に見る

今回の結果では、`tdoa_corrected_ms` は左側キーで +1.0ms 付近まで大きくなるものがあり、マイク間21cmから見た物理上限 ±0.612ms を超えています。

そのため、今回の評価では **`tdoa_raw_ms_CH2_minus_CH1` を主に見る**のが自然です。

`tdoa_corrected_ms` は参考値として残していますが、今回のデータでは補正が過剰になっている可能性があります。

## 結果の読み取り

全体としては、かなり期待に近い傾向が出ています。

```text
Q/W/E/R/T : 正の値 → 左側キーとして検出
Y         : 0付近 → 中央付近として検出
U/I/O    : 負の値 → 右側キーとして検出
P         : 1回は右側、1回は外れ
```

特に、左から右へ移動するにつれて、TDoAの値が

```text
正 → 0付近 → 負
```

へ変化している点は重要です。

## 主な結果

### 左側キー

Q, W, E, R, T はすべて raw Δt が正の値になりました。

これは、CH1、つまり左側マイクの方が早く音を拾っていることを意味します。

```text
Q平均: +0.446 ms
W平均: +0.525 ms
E平均: +0.361 ms
R平均: +0.305 ms
T平均: +0.282 ms
```

QとWは順序が少し逆転していますが、どちらも左側としては明確に出ています。

### 中央付近

Yは2回とも0付近でした。

```text
Y: +0.019 ms, +0.048 ms
平均: +0.034 ms
```

これは、Yが左右マイクの中央寄りにあることと整合します。

### 右側キー

U, I, O はすべて raw Δt が負の値になりました。

```text
U平均: -0.173 ms
I平均: -0.300 ms
O平均: -0.342 ms
```

右側へ行くほど、CH2、つまり右側マイクが先に拾う傾向が出ています。

### Pについて

Pは2回中1回が外れました。

```text
P 1回目: +0.586 ms, best_corr = 0.279
P 2回目: -0.457 ms, best_corr = 0.611
```

P 2回目は期待通り、右側として検出されています。

一方、P 1回目は本来右側のはずですが、左側のような値になっています。ただし `best_corr = 0.279` と低いため、TDoA推定に失敗している可能性が高いです。

## best_corrによる信頼度

今回、`best_corr` が低かったものは以下です。

| キー | 回 | raw Δt[ms] | best_corr | コメント |
|---|---:|---:|---:|---|
| Q | 2 | +0.277 | 0.297 | 左右判定は合っているが信頼度は低め |
| P | 1 | +0.586 | 0.279 | 期待と逆方向。外れ値の可能性が高い |

目安として、`best_corr < 0.45` のデータは、TDoA値を信用しすぎない方がよさそうです。

学習に入れる場合は、以下のように扱うのが安全です。

```python
if best_corr < 0.45:
    tdoa_feature_ms = 0.0
    tdoa_reliable = 0
else:
    tdoa_feature_ms = tdoa_raw_ms
    tdoa_reliable = 1
```

## 結論

今回の結果では、TDoAはかなり有望です。

単独でキー名を完全に識別する特徴量としてはまだ不足がありますが、少なくとも以下の用途には使えそうです。

- 左側キーか右側キーかの判定
- 音響特徴量だけでは近いキーを区別しにくい場合の補助特徴量
- SVMなどに入れる位置情報特徴量

ただし、P 1回目のように外れるケースがあるため、TDoAをそのまま過信するのは危険です。

実運用では、以下の特徴量をセットで使うのが良いです。

```text
tdoa_feature_ms
tdoa_reliable
best_corr
energy_CH1_over_CH2_dB
```

現時点の判断としては、**TDoAは主特徴量ではなく、キーの左右位置を補助する特徴量として使うのが妥当**です。
